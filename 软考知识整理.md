## 1、码制


### 表示
| 码制 | 定点整数                  | 定点小数                        | 数码个数 | 用途 |
| :- | :-------------------- | :-------------------------- | :--- | :--- |
| 原码 | -(2ⁿ⁻¹-1) ~ +(2ⁿ⁻¹-1) | -(1-2⁻⁽ⁿ⁻¹⁾) ~ +(1-2⁻⁽ⁿ⁻¹⁾) | 2ⁿ-1 | 简单表示，乘除法 |
| 反码 | -(2ⁿ⁻¹-1) ~ +(2ⁿ⁻¹-1) | -(1-2⁻⁽ⁿ⁻¹⁾) ~ +(1-2⁻⁽ⁿ⁻¹⁾) | 2ⁿ-1 | 原码与补码的中间过渡 |
| 补码 | -2ⁿ⁻¹ ~ +(2ⁿ⁻¹-1)     | -1 ~ +(1-2⁻⁽ⁿ⁻¹⁾)           | 2ⁿ   | 计算机内部存储和运算整数（最常用） |
| 移码 | -2ⁿ⁻¹ ~ +(2ⁿ⁻¹-1)     | -1 ~ +(1-2⁻⁽ⁿ⁻¹⁾)           | 2ⁿ   | 浮点数的阶码表示，便于比较大小 |

计算机内部用补码存整数，因为加减法统一、0 唯一。例：-3 的补码计算过程：原码 10000011 → 反码 11111100 → 补码 11111101。
### 考点
- 补码范围不对称：最小负数的绝对值比最大正数大1
如8位补码：-128 ~ +127
- 补码中 -2ⁿ⁻¹ 没有原码和反码
10000000（8位）在补码中表示-128，在原码中是非法的 
- 移码与补码的关系：符号位相反，数值位相同 
- 定点小数 -1 的补码：1.000...0（纯小数，整数部分为1表示负数）
## 2、浮点数
### 格式
阶码决定范围，尾数决定精度。越长，范围越大，精度越高。
### 计算过程
1. 解析浮点数：从存储的二进制位中提取符号位、指数位和尾数位。
2. 处理指数：对于加减法，需要对齐指数；对于乘除法，指数相加或相减。
3. 处理尾数：将尾数对齐或相乘，并考虑进位或舍入。
4. 规格化：确保计算结果符合 IEEE 754 规范。
5. 编码为浮点数：将最终结果重新组织成二进制形式。

**过程：对阶 -> 尾数计算 -> 格式化**

**对阶：小数向大数看齐，尾数右移**

- **理论公式**：建立时间 + (指令条数 - 1) × 流水线周期

## 12、加密算法分类、优缺点和应用场景

加密算法主要分为三类：

- **对称加密**：使用单一密钥进行加密和解密，速度快、资源消耗少，适合大数据量加密，但密钥管理复杂，常用于文件加密和磁盘加密。
- **非对称加密**：使用公钥加密、私钥解密，安全性高，但计算速度慢，常与对称加密结合用于数字签名和安全通信，如 HTTPS 协议。
- **哈希算法**：生成固定长度的摘要。

## 13、HTTPS 通信加解密流程

HTTPS 通信的加解密流程主要分为两个阶段：**握手阶段**和**数据传输阶段**。

在握手阶段，客户端向服务器发送请求，服务器回应数字证书和公钥。客户端验证证书后，生成随机对称密钥，用服务器公钥加密后发送给服务器。服务器用私钥解密得到对称密钥。

#### 解题技巧
关键路径法口诀：正推取最大，逆推取最小，关键路径工期定
| 步骤 | 操作                   |
| :- | :------------------- |
| 1  | 列出所有从START到FINISH的路径 |
| 2  | 计算每条路径的总工期           |
| 3  | **最大工期**即为项目最短完成时间   |
| 4  | 工期等于最大值的即为**关键路径**   |

逆推取最小用于计算节点的最迟开始时间（LS）或最迟完成时间（LF），主要用于：
| 计算方向              | 目的            | 计算规则     | 结果          |
| :---------------- | :------------ | :------- | :---------- |
| **正推**（从START开始）  | 求**最早开始时间ES** | **取最大值** | 确定关键路径工期    |
| **逆推**（从FINISH倒推） | 求**最迟开始时间LS** | **取最小值** | 确定浮动时间/松弛时间 |

二、什么情况下需要逆推取最小？
1. 计算最迟时间（确定浮动时间）
   当需要计算：
- 最迟完成时间（LF）
- 最迟开始时间（LS）= LF - 活动持续时间
- 总浮动时间（TF）= LS - ES 或 LF - EF
2. 判断关键活动（TF=0）
3. 关键活动：ES = LS 且 EF = LF（浮动时间为0）
   非关键活动：有浮动时间，可以延迟而不影响总工期

### 图
#### 概念
有向图的深度优先遍历（DFS）和广度优先遍历（BFS）是两种基本的图遍历算法，用于系统地访问图中的所有顶点。
#### 过程对比
【DFS遍历过程】          【BFS遍历过程】

    A                    A ←──── 第1层
    ↓                   / \
    B                   B   C ←── 第2层
    ↓ ↘                 ↓ ↓
    D   E               D   E ←── 第3层
    （深入到底再回溯）    （层层扩展）

DFS顺序：A→B→D→E→C     BFS顺序：A→B→C→D→E
（先走B分支到底，再回溯）

#### 示例
有向图：
A → B → D
↓   ↓
C → E

DFS遍历（从A开始）：
A → B → D（D无出边，回溯）
→ E（E无出边，回溯）
→ C（C已访问过E，跳过）

结果：A, B, D, E, C 或 A, B, E, D, C（取决于邻接点顺序）

有向图：
A → B → D
↓   ↓
C → E

BFS遍历（从A开始）：
第1层：A
第2层：A的邻接点 → B, C
第3层：B的邻接点 → D, E（C的邻接点E已在队列或已访问）

结果：A, B, C, D, E


#### DFS 和 BFS对比

| 对比项       | DFS（深度优先）                       | BFS（广度优先）      |
| :-------- |:--------------------------------| :------------- |
| **核心思想**  | 从起始顶点出发，尽可能深地探索图的分支，直到无法继续才回溯   | 从起始顶点出发，先访问所有邻接顶点，再依次访问这些邻接顶点的邻接顶点，层层扩展 |
| **数据结构**  | 栈（递归/显式栈）                       | 队列             |
| **遍历顺序**  | 先深入，后回溯                         | 层层扩展，先近后远      |
| **空间复杂度** | O(V)（递归栈深度）                     | O(V)（队列最大宽度）   |
| **时间复杂度** | O(V + E)                        | O(V + E)       |
| **最短路径**  | ❌ 不能保证                          | ✅ **无权图最短路径**  |
| **适用场景**  | 拓扑排序、连通性、回溯问题                   | 最短路径、层次遍历、社交网络 |
| **有向图特点** | 需考虑边的方向，可能无法访问所有顶点              | 同样受出边方向限制      |

### 判断DFS/BFS序列的技巧
第一步：画邻接表

v1: [v2, v3]   ← 出边指向
v2: [v4, v5]
v3: [v4]
v4: [v5]
v5: [v2]

第二步：DFS快速判断法 (核心口诀：能深则深，走不通再回溯)
快速检验：相邻两点是否有边？
| 序列               | 相邻检查               | 结果      |
| :--------------- | :----------------- | :------ |
| ① v1→v2→v3       | v2→v3？❌ 无此边        | **秒排除** |
| ② v1→v3→v4→v5→v2 | 每步都有边✓             | 合法      |
| ③ v1→v3→v2       | v3→v2？❌ 无此边        | **秒排除** |
| ④ v1→v2→v4→v5→v3 | v5→v3？❌ 但可回溯到v1→v3 | 合法      |
DFS回溯判断：序列中后面出现的点，前面某点的邻接点
④ v1,v2,v4,v5,v3
↑_________↑
v5走不通，回溯到v1，v1还有v3未访问

第三步：BFS快速判断法(核心口诀：层层扩展，同层优先)
快速检验：是否按层遍历？
第1层: v1
第2层: v2, v3（v1的邻接点，顺序可换）
第3层: v4, v5（v2的邻接点 + v3的邻接点去重）

① v1, v2, v3, v4, v5 ✓ 按层
② v1, v3, v4...      ✗ v4是第3层，v2是第2层，v2应在v4前
③ v1, v3, v2, v4, v5 ✓ 按层（第2层顺序v3,v2）
④ v1, v2, v4...      ✗ v4是第3层，v3是第2层，v3应在v4前

### 贪心算法
#### 核心思想
特点：局部最优 → 期望全局最优（但不一定能得到全局最优）
┌─────────────────────────────────────────┐
│           贪心算法（Greedy Algorithm）      │
├─────────────────────────────────────────┤
│  1. 将问题分解为若干子问题                 │
│  2. 对每个子问题求局部最优解               │
│  3. 把局部最优解合成原问题的一个解           │
│                                         │
│  关键：每一步都选当前看起来最好的，不回溯      │
└─────────────────────────────────────────┘
#### 经典示例：找零钱问题
问题
假设有硬币：1元、5元、10元，要凑出 36元，最少需要几枚？
贪心策略
每次选面值最大的硬币
目标：36元

第1步：选10元（最大）→ 剩余 36-10=26元，用了1枚
第2步：选10元（最大）→ 剩余 26-10=16元，用了2枚
第3步：选10元（最大）→ 剩余 16-10=6元，用了3枚
第4步：选5元（最大）→ 剩余 6-5=1元，用了4枚
第5步：选1元（最大）→ 剩余 0元，用了5枚

结果：10+10+10+5+1 = 36，共5枚 ✓

| 排序算法     | 平均时间         | 最好         | 最坏         | 说明 |
| :------- | :----------- | :---------- | :---------- | :--- |
| **快速排序** | **O(n log n)** | O(n log n)  | O(n²)       | 最常用，但最坏情况退化 |
| 归并排序     | O(n log n)   | O(n log n)  | O(n log n)  | 稳定但需O(n)额外空间 |
| 堆排序      | O(n log n)   | O(n log n)  | O(n log n)  | 原地排序，无最坏退化 |
| 冒泡排序     | O(n²)        | O(n)        | O(n²)       | 最好情况O(n)已有序 |
| 简单选择排序   | O(n²)        | O(n²)       | O(n²)       | 始终O(n²)，不稳定 |
### 空间复杂度
| 排序算法     | 辅助空间复杂度         | 说明                      |
| :------- | :-------------- | :---------------------- |
| **归并排序** | **O(n)**        | ✅ **最多**，需要额外数组存储合并结果   |
| 快速排序     | O(log n) ~ O(n) | 递归栈空间，平均O(log n)，最坏O(n) |
| 堆排序      | O(1)            | 原地排序，仅需常数额外空间           |
| 冒泡排序     | O(1)            | 原地排序，仅需临时变量             |
### 稳定性
| 排序算法     | 稳定性       | 判断                |
| :------- | :-------- | :---------------- |
| **冒泡排序** | **稳定** ✅  | 相等元素不交换，相对位置不变    |
| 快速排序     | **不稳定** ❌ | 分区交换可能改变相等元素相对位置  |
| 堆排序      | **不稳定** ❌ | 建堆和调整过程会改变元素相对位置  |
| 简单选择排序   | **不稳定** ❌ | 选择最小元素交换时可能改变相对位置 |

### 增量模型
#### 定义
┌─────────────────────────────────────────┐
│           增量模型（Incremental Model）    │
├─────────────────────────────────────────┤
│                                         │
│  将系统分解为多个增量，分阶段开发和交付        │
│                                         │
│  第1增量 ──→ 第2增量 ──→ 第3增量 ──→ ...   │
│  [核心功能]   [扩展功能]   [完善功能]       │
│     ↓          ↓          ↓             │
│   可用产品    功能增强    完整系统         │
│                                         │
│  每个增量都是一个可运行的版本              │
└─────────────────────────────────────────┘

增量模型 vs 瀑布模型
| 瀑布模型      | 增量模型         |
| :-------- | :----------- |
| 一次性完成所有阶段 | 分多次迭代，每次一个增量 |
| 最后才交付产品   | 早期就交付可用版本    |
| 需求变更困难    | 适应需求变化       |

#### 优缺点
- 优点：快速交付、用户早期参与、风险分散
- 缺点：设计难度大、架构要求高、接口复杂

| 优点           | 解释          |
| :----------- | :---------- |
| 快速交付可用产品     | 核心功能优先开发    |
| 降低风险         | 早期发现问题      |
| 用户早期参与       | 及时反馈，减少需求偏差 |
| 优先级高的功能测试更充分 | 多次迭代中使用     |
| 适应需求变化       | 后续增量可调整     |


### 软件设计原则
| 原则   | 正确表述          |
| :--- | :------------ |
| 模块化  | 将系统划分为相对独立的模块 |
| 高内聚  | 模块内部元素联系紧密    |
| 低耦合  | 模块之间依赖尽可能少    |
| 模块大小 | **适中**，不是越小越好 |
| 扇入扇出 | 合理控制，避免过高或过低  |

- 扇入： 调用该模块的上级模块个数
- 扇出：该模块调用的下级模块个数
